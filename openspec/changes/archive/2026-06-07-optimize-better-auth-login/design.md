## Context

当前项目已完成 better-auth 的基础配置：
- 服务端配置了 Prisma 数据库适配器连接 PostgreSQL
- 启用了邮箱密码认证和微信社交登录
- 数据库 schema 包含 User、Session、Account 和 Verification 表
- API 路由已设置为 `/api/auth/[...all]`

但缺少完整的用户界面和客户端集成：
- 没有登录注册页面
- 客户端没有会话管理逻辑
- 缺少安全配置（CSRF、速率限制）
- 没有路由保护机制

技术栈约束：
- Next.js 16.2.7（App Router）
- React 19.2.4
- better-auth 1.6.14
- PostgreSQL + Prisma
- 遵循 Jo's Thought Imprint 编码原则（避免过度抽象、优先内联简单逻辑）

## Goals / Non-Goals

**Goals:**
- 实现完整的邮箱密码和微信登录/注册流程
- 提供安全的会话管理和持久化
- 保护需要认证的路由和 API
- 配置生产级别的安全措施（CSRF、速率限制、安全 cookie）
- 邮箱验证和密码重置功能

**Non-Goals:**
- 多因素认证（2FA/TOTP）—— 后续迭代考虑
- 角色权限系统（RBAC）—— 当前仅验证身份，不区分权限
- 账号合并逻辑 —— 邮箱账号和微信账号独立，不自动合并
- 第三方身份提供商集成（除微信外）
- 企业级审计日志系统 —— 仅记录基本审计信息到 Session 表

## Decisions

### 决策 1: 使用 better-auth 的内置功能而非自定义实现
**选择**: 使用 better-auth 提供的邮箱验证、密码重置、CSRF 保护等插件
**原因**: 
- better-auth 已包含生产级实现，减少安全漏洞风险
- 避免重新发明轮子，符合 Jo's 原则不做不必要的抽象
**替代方案**: 自行实现验证令牌生成和邮件发送逻辑 —— 增加维护成本且容易出错

### 决策 2: 客户端使用 better-auth/react 的 createAuthClient
**选择**: 使用官方 React 客户端而非手动管理会话
**原因**:
- 自动处理会话刷新、跨标签页同步
- 提供 React hooks（useSession）与组件生态无缝集成
- 内置 CSRF 令牌管理
**替代方案**: 使用 Context API 自建会话管理 —— 需要处理复杂的令牌刷新和同步逻辑

### 决策 3: 中间件实现路由保护
**选择**: 在 `src/proxy.ts` 中使用 Next.js proxy 验证会话
**原因**:
- 服务端验证，无法被客户端绕过
- 在请求到达页面前拦截，性能更好
- 可以统一管理受保护路由列表
**替代方案**: 在每个页面组件中单独检查会话 —— 容易遗漏且代码重复

### 决策 4: 邮件发送使用环境变量配置的 SMTP（已实现为 Resend）
**选择**: better-auth 的 `sendEmail` 回调中使用 Resend API
**原因**:
- Resend 提供简单的 API 和良好的送达率
- 开发阶段使用测试域名快速验证功能
- 通过环境变量切换测试和生产配置
**替代方案**: 使用 nodemailer + SMTP —— 配置复杂且送达率依赖 SMTP 服务商

### 决策 8: 注册流程采用"先验证后发送"策略
**选择**: 前端验证邮箱格式、密码强度、邮箱是否已存在，验证全部通过后才调用注册 API
**原因**:
- 减少无效请求，避免向无效邮箱发送邮件
- 提前检测邮箱重复，提升用户体验（无需等待邮件才发现已注册）
- 降低邮件服务成本和被标记为垃圾邮件的风险
**替代方案**: 先调用 API 后再验证 —— 会导致向无效邮箱发送邮件，浪费资源

### 决策 9: 邮件发送频率限制（5分钟冷却期）
**选择**: 同一邮箱地址5分钟内只能发送一次验证邮件
**原因**:
- 防止用户恶意重复请求造成邮件轰炸
- 保护邮件服务配额和发送声誉
- 5分钟足够用户检查邮箱（包括垃圾箱）
**替代方案**: 无限制发送 —— 容易被滥用；更长冷却期（如10分钟）—— 影响用户体验
**实现**: 在服务端验证上次发送时间戳，存储在 Verification 表中

### 决策 10: 验证成功后重定向到登录页而非直接登录
**选择**: 邮箱验证成功后跳转到 `/login` 而非 `/` 或自动登录
**原因**:
- 分离注册和登录流程，符合传统 Web 应用模式
- 用户可以在登录时选择"记住我"等选项
- 避免自动登录带来的安全风险（验证链接可能被转发）
- 给用户明确的操作提示（"验证成功，请登录"）
**替代方案**: 
- 验证后自动登录 —— 存在安全风险（链接泄露）
- 跳转到首页 —— 用户需要再次寻找登录入口，体验差

### 决策 11: 注册成功后跳转到首页并提示
**选择**: 注册 API 调用成功后跳转到 `/` 并显示提示"注册成功，请查收邮件并验证后登录"
**原因**:
- 用户可以看到应用主界面，了解产品功能
- 明确告知下一步操作（查收邮件 → 验证 → 登录）
- 避免停留在注册页造成困惑
**替代方案**: 停留在注册页 —— 用户不清楚是否成功；直接跳转登录页 —— 用户还未验证邮箱

### 决策 12: 集成人机验证防止机器人注册
**选择**: 注册表单集成 CAPTCHA（如 Google reCAPTCHA v3 或 Cloudflare Turnstile）
**原因**:
- 防止机器人批量注册消耗系统资源
- 保护邮件服务配额不被滥用
- 提升账号质量，减少垃圾账号
- reCAPTCHA v3 无需用户交互，体验友好
**替代方案**: 
- 无验证 —— 容易被批量注册攻击
- 传统图片验证码 —— 用户体验差，可访问性不佳
- 邮箱验证作为唯一防护 —— 无法防止批量发送垃圾邮件
**实现**: 前端集成 CAPTCHA 组件，服务端验证 token，验证失败拒绝注册

### 决策 11: 首页显示登录/注册入口和用户组件
**选择**: 首页右上角根据会话状态动态显示：
- 未登录: 显示"登录"和"注册"按钮
- 已登录: 显示用户头像/名称和"退出登录"按钮
**原因**:
- 用户无需记忆特定 URL 即可访问认证功能
- 登录状态一目了然，符合用户直觉
- 退出登录功能易于访问
**替代方案**: 隐藏在菜单或独立页面 —— 不符合 Web 应用常见模式

### 决策 5: 页面布局使用 Route Groups
**选择**: 
- `(auth)` 组: 登录、注册页面（无需认证，简洁布局）
- `(protected)` 组: 个人中心等需要认证的页面（包含导航栏）
**原因**:
- Next.js App Router 推荐模式，清晰分离不同布局
- 避免在根 layout 中做复杂的条件渲染
**替代方案**: 单一 layout 通过条件判断渲染 —— 逻辑复杂且难以维护

### 决策 6: 微信登录使用服务端回调处理
**选择**: 微信授权回调到 `/api/auth/callback/wechat`，better-auth 自动处理
**原因**:
- 敏感凭证（client secret）仅在服务端使用
- better-auth 内置 OAuth 流程处理，包括 state 参数验证
**替代方案**: 客户端直接处理回调 —— 安全风险高

### 决策 7: 密码验证规则最低8字符
**选择**: 要求密码至少8个字符，包含字母和数字
**原因**:
- 平衡安全性和用户体验
- 符合一般 Web 应用的安全标准
**替代方案**: 更严格规则（特殊字符、大小写）—— 可能降低用户体验

## Risks / Trade-offs

### 风险 1: 邮件发送失败导致用户无法验证
**缓解**: 
- 提供"重新发送验证邮件"功能
- 日志记录邮件发送失败，监控邮件服务可用性
- 考虑添加备用邮件服务商

### 风险 2: 微信接口限流或不可用
**缓解**:
- 捕获微信 API 错误并向用户展示友好提示
- 提供邮箱登录作为备用方式
- 监控微信登录成功率

### 风险 3: 会话劫持攻击
**缓解**:
- 使用 httpOnly + secure + sameSite cookie
- 记录 IP 地址和 User-Agent 到 Session 表
- 实施速率限制防止暴力破解

### 风险 4: CSRF 令牌被绕过
**缓解**:
- better-auth 自动管理 CSRF 令牌
- 所有状态变更请求（POST/PUT/DELETE）验证令牌
- 仅在同源请求中自动附加令牌

### 权衡 1: 未实现账号合并
**影响**: 用户用邮箱注册后无法直接关联微信，需要手动在设置中关联
**理由**: 自动合并需要复杂的身份识别逻辑（邮箱匹配、用户确认），当前阶段过度复杂

### 权衡 2: 速率限制基于 IP 地址
**影响**: NAT 环境下多用户共享 IP 可能被误伤
**理由**: 实现简单且覆盖大部分场景，后续可引入更精细的限制策略（基于 fingerprint）

## Migration Plan

### 部署步骤
1. 确保环境变量配置完整：
   - `DATABASE_URL`: PostgreSQL 连接字符串
   - `BETTER_AUTH_SECRET`: 会话加密密钥（生产环境必须随机生成）
   - `BETTER_AUTH_URL`: 应用公开访问 URL（用于回调）
   - `WECHAT_CLIENT_ID` / `WECHAT_CLIENT_SECRET`: 微信开放平台凭证
   - SMTP 配置: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`

2. 运行数据库迁移：
   ```bash
   npx prisma migrate deploy
   ```
   （当前 schema 已包含所需表，无需新迁移）

3. 构建和部署应用：
   ```bash
   npm run build
   npm run start
   ```

4. 验证关键功能：
   - 注册新账号并接收验证邮件
   - 邮箱密码登录
   - 微信扫码登录
   - 访问受保护路由触发重定向

### 回滚策略
- 如果邮件发送失败：临时禁用邮箱验证要求（修改 `emailAndPassword.requireEmailVerification`）
- 如果微信登录异常：移除 `socialProviders.wechat` 配置，暂时仅支持邮箱登录
- 数据库层无破坏性变更，回滚代码即可

## Open Questions

1. **邮件服务商选择**: 使用自建 SMTP、阿里云邮件推送还是 SendGrid？
   - 建议: 开发环境使用 Mailtrap 测试，生产环境根据成本和送达率选择

2. **速率限制的具体数值**: 当前设计为1分钟5次登录尝试，是否合理？
   - 建议: 初期上线后根据实际日志调整，可能需要针对注册和登录设置不同限制

3. **邮箱验证是否强制**: 未验证邮箱的用户是否允许登录？
   - 建议: 初期允许登录但限制部分功能（如发布内容），后续根据业务需求调整

4. **微信登录的默认用户名**: 微信昵称可能包含特殊字符或 emoji，如何处理？
   - 建议: 存储原始昵称到 `name` 字段，数据库已设置为 `String` 类型支持 Unicode

5. **会话过期时间**: 当前未显式配置，使用 better-auth 默认值（7天），是否需要调整？
   - 建议: 保持默认，后续根据用户反馈调整
